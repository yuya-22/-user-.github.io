## WBS

WBSとは、プロジェクト目標を達成し、必要な 成果物を作成するために、プロジェクトチームが行う作業を、成果物を中心に断層的に細分化したもの

### メリット
スコープが明確になる

作業を洗い出せる

全体管理と作業計画が明確化される

プロジェクト実施時はWBSに則り実行するのみ

### WBSの作り方

１，スコープを明確にする

２，大きな作業をグルーピングする

３，グルーピングした作業の相互関係を考える

４，各グループの作業を洗い出す

## コーディング

コードは書くよりも読まれることの方が多い

よって他人や未来の自分が読みやすいコードを書かなくてはならない

### PEP8

これはpythonのコードを読みやすいようにするために定められたルールをまとめたものである

### flake8

flake8をパイソンのコードに使うと、PEP8に従っていないところをエラーとして列挙してくれる

これによりエラー個所を見つけ修正すると読みやすいコードが出来上がる。

## バージョン管理

ファイルを誰が、いつ、どのように変更したかを管理する

集中管理型や分散管理型などがある

### 集中管理型

同時編集をするとコンフリクトが発生しやすい

同期に時間がかかる

### 分散管理型

リモートリポジトリへのアクセス頻度が低い

障害に頑健

## git
gitとは、分散管理型のバージョン管理システム

変更履歴が残る

偏向した箇所に戻ることができる

他人と共同編集できる

## CI/CD

### CI
コード変更を共有リポジトリに頻繁に統合するプロセス

自動テストとビルドを定期的に実行し、バグの早期発見と修正を可能に

### CD
コード変更をテスト環境や本番環境に自動的にデプロイするプロセス

自動デプロイメントを組み込み、手動操作によるデプロイの必要性を排除

CDにより、ユーザーフィードバックを迅速に反映可能

### CI/CDパイプラインの基本的なステップ

1,コード変更をトリガーにワークフローを起動

2,ソースコードをコンパイル

３，自動テストを実行

４，テスト済みコードを本番環境にデプロイ

５，デプロイされたアプリケーションの動作を確認

6,本番環境での継続的な監視
